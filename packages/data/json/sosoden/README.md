# 조조전 시스템 데이터 셋 (sosoden/)

> 코에이 《삼국지 조조전》(1998) 원본에서 추출한 **v1 전투 시스템 원천** 데이터.
> CLAUDE.md §2-9(2026-06-13 확정): **맵=영걸전 / 시스템=조조전**. 병종·능력치·책략·아이템·전투공식·일기토는 이 셋을 원천으로 런타임 데이터(packages/data/json/)에 반영한다. 맵/지형은 영걸전 유지.
> ⏳ 능력치 필드 순서·chunk2/3/4 병종표 해독 후 런타임 데이터로 확정 (아래 "다음 단계").

추출 경위·포맷 분석: [docs/reference/sosoden-source-analysis.md](../../../../docs/reference/sosoden-source-analysis.md)
재생성: `tools/hero-extract/extract_sosoden_data.py`, `eex_extract.py`

## ⚠️ IP / 커밋 주의

- **수치 테이블**(스탯·power·MP 등)은 사실 데이터(저작권 비보호) — 영걸전 셋과 동일 취급.
- **scripts/ 의 대사 텍스트는 코에이의 저작 대사 원문** — 내부 분석/레퍼런스 한정. 게임에 원문 그대로 싣지 않는다. 저장소 커밋 여부는 길중 판단 필요.

## 파일

### `generals.json` — 장수 512명 (DATA.E5 chunk 0, 32B/레코드)
| 필드 | 의미 | 신뢰도 |
|---|---|---|
| `name` | 장수명 (CP949) | ✅ 확정 |
| `classId` / `className` | 병종 번호 (JOB.TXT 번호표와 일치 — 조조=군주계, 관우=기병계) | ✅ 검증 |
| `graphicId` | 얼굴/그래픽 id | ✅ |
| `stats_18_22` | 핵심 능력치 5바이트 (무력/지력/통솔/순발/운 계열, **순서 미확정**) | ⚠️ 추정 |
| `val23` | 사기/HP 계열 (~100) | ⚠️ 추정 |
| `level` | 레벨(기본값) | ⚠️ 추정 |
| `val29` | 진영/플래그 | ⚠️ 추정 |
| `_raw` | 32바이트 hex 원본 (재해석용) | — |

### `weapons.json` — 무기/아이템 104종 (DATA.E5 chunk 1, 25B/레코드)
| 필드 | 의미 | 신뢰도 |
|---|---|---|
| `name` | 아이템명 | ✅ |
| `power` | 위력 (단검5<대검10<강검30 순서로 검증) | ✅ 검증 |
| `markerAt` / `preMarker` / `fields` | 0xff 마커 위치 + 전/후 바이트 | ⚠️ 구조 |
| `_raw` | 25바이트 hex | — |

### `strategies.json` — 책략 73종 (DATA.E5 chunk 5, 70B/레코드)
화공(초열·업화·화룡)·수공(탁류·해일)·풍(선풍·풍룡)·지(낙석·지진)·정신계(유혹·현혹·둔병) 등.
| 필드 | 의미 | 신뢰도 |
|---|---|---|
| `name` | 책략명 | ✅ |
| `mp` | MP 소모 (추정) | ⚠️ |
| `val15` | 위력/레벨 (추정) | ⚠️ |
| `tail` | 레벨별 데미지 삼중값으로 보이는 후반부 (off 40~) | ⚠️ 구조 |
| `_raw` | 70바이트 hex | — |

### `classTablesRaw.json` — 병종/성장 수치표 (DATA.E5 chunk 2,3,4)
순수 수치 테이블(한글 없음). 병종 클래스 정의·성장률·책략 해금 후보. stride 추정값 + `_raw` 동봉, 필드 미해독.

### `scripts/*.json` — EEX 대사 스크립트 117개 (+ `_index.json`)
스테이지별 시나리오(S_xx) + 이벤트/인트로(R_xx)의 대사·내레이션 텍스트 레이어.
- `lines[]`: `{off, enc(cp949|sjis), text}` — 화자명+대사가 순서대로 (예: R_00 = 허소의 조조 평 "乱世の姦雄、治世の能臣").
- **미포함**: 구조화된 이벤트/일기토 발동 트리거. EEX의 opcode 섹션(텍스트 앞)에 인코딩돼 있으며, 이를 우리 stage `events[]` 스키마로 디컴파일하는 건 별도 후속 작업.
