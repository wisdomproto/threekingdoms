## 2. BAKDATA.R3 완전 구조 (19,328 bytes)

신규 게임 초기 상태의 마스터 데이터. 에디터(README.TXT)가 "능력치/이름/얼굴/겉모습 + 아이템 숨김 수치"를 편집하는 대상 파일이 바로 이것.

| 구획 | 오프셋 | 레코드 | 개수 | 내용 |
|---|---|---|---|---|
| A | `0x0000` | 13B | 256 | NPC 표시명 (노인/농민/상인/미녀…) + 얼굴 번호 |
| B | `0x0D00` | 16B | 63 | **아이템 정의** (§4) |
| C | `0x1100` | 21B | 384 | **장수 마스터** (§3) |
| D | `0x3080` | 18B | 384 | **초기 부대 상태** (C와 1:1 대응) |

### 21B 장수 레코드 (구획 C)
```
+0  char[6]  이름 (CP949, null 패딩)
+6  byte[8]  미해독 (자체 폰트 렌더링 코드 추정)
+14 u16 LE   얼굴 번호 (FACEDAT.R3 인덱스 0~239)
+16 u8       미상 (0~28, 전투 대화 소형 초상 또는 외형 그룹 추정)
+17 u8       통솔력
+18 u8       무력
+19 u8       지력
+20 u8       겉모습(전투 스프라이트/색상 복합 코드 추정)
```

### 18B 초기 부대 레코드 (구획 D) — 중국 커뮤니티 세이브 구조와 동일 레이아웃
```
+0  u8       소속 군단 (0x80=유비군, 0x81=조조 … +13=공융, 0x8E=이탈/전사)
+1  u8       편제 플래그 (미상)
+2  u16      (0)
+4  u8       사기 (초기 전원 100)
+5  u16 LE   병력
+7  u8       병종 코드 (§5의 0x00~0x12)
+8  u16 LE   레벨
+10 byte[8]  소지 아이템 슬롯 8칸 (0xFF=빈칸, 적장 소지품 = 격파 드랍 후보)
```
세이브 파일(`MSAVE0~4.R3S`=무장, `ESAVE0~4.R3S`=스테이지 진행)도 동일 레코드 구조 (출처: CSDN, §11).

## 4. 아이템 63종 + 숨김 수치

레코드: `이름 char[13]` + `b13(효과치)` + `b14(% 보정치)` + `b15(분류)`.

**분류(b15)**: 0=무기 / 1=보물(지속 패시브) / 2=공격 소모품 / 3=회복·보조 소모품(+이벤트템) / 4=말 / 5=병법서
**b13**: 소모품의 효과량 (회복량/데미지/사기치). 255 = 비소모(장비/보물).
**b14**: 장비의 **% 가산 보정** — 무기=공격력, 병법서=책략. 웹 자료의 "삼첨도 110%, 패자검 124%, 손자병법서 122%"와 정확히 일치 (b14=10, 24, 22). 말은 등급치(1~3, 이동력 보너스 추정).

### 무기 (공격력 +b14%)
| 무기 | 보정 | 무기 | 보정 |
|---|---|---|---|
| 패자검 | **+24%** | 의천검 | +14% |
| 영걸검 | +22% | 청룡언월도 | +12% |
| 청강검 | +20% | 칠성검/삼첨도/사모 | +10% |
| 방천화극 | +18% | 자웅일대검 | +16% |

### 병법서 (책략 위력 +b14%) / 말
손자의병법서 +22% / 오자의병법서 +20% / 육도·삼략·맹덕신서 +15%
조황비전(1) < 적로(2) < 적토마(3)

### 소모품 효과치 (b13)
- 회복(병력): 콩 10 / 보리 20 / 쌀 40 — 사기: 술 5 / 특급주 10 / 빼갈 20
- 회복(책략치): 상약 30 / 한방약 60 / 차 120
- 공격: 폭탄 50 / 화계서(초열5→화룡10→업화20) / 수계서(소용돌이8→탁류16→해일32) / 산계서(낙석10→설붕20→산사태40) / 날씨(농무5→기상10→호우20)
- 증원·사기: 원대서 50 / 원부서 100 / 원군서 200, 평기서 25 / 활기서 50 / 용기서 100
- 보물(분류1)의 b13: 장창30/보병차60/연노35/투석기70/마개40/근위개80/무술교본20/의협교본40 — 병종별 강화 보물의 효과치로 추정

전체 테이블: `tools/hero-extract/out/items_table.md`

- **승급 체계**: 보병(단병→장병→전차), 궁병(궁병→연노병→발석차), 기병(경기병→중기병→친위대), 적병(산적→흉적→의적) — 3단계

- 지형 종류 (MAIN.EXE): 평지/초원/산지/개울/다리/성벽/낭떠러지/황무지/울타리/병영/군량고/보물창고/화염/탁류 (복각판 기준 20종: +삼림/성/촌락/성문/녹채/가옥)

## 6. 전투 공식 (웹 커뮤니티 역산 자료 — 한·일 자료 일치, §11 출처)

```
공격력 = (병종공격력 + 사기 + 보정무력) × (10 + 레벨) ÷ 10 × 아이템보정%
방어력 = (병종방어력 + 사기 + 보정통솔) × (10 + 레벨) ÷ 10 × 아이템보정%

보정능력치 = 4000 ÷ (140 − 원능력치)
  → 능력치 100 = 100, 90 = 80, 80 = 67, 60 = 50, 40 = 40
  → 80 이상 구간에서 가치가 비선형 급증 (명장 희소성의 수학적 근거)

데미지 = (공격력 − 방어력 × 상성계수 ÷ 2) × 지형보정%
  상성계수: 유리 0.75 / 보통 1.0 / 불리 1.25
  지형 데미지 경감: 초원·마을 5% / 병영 10% / 숲 20% / 성·산 30%

책략치(MP) = (레벨 + 10) × 지력 ÷ 40
사기: 기본 100, 피격 시 감소. 공/방 양쪽에 직접 가산 → 사기 관리가 핵심 메커니즘
경험치 = 6 + 레벨차×2 (+퇴각시 32, +총대장 16). 통상 공격은 반드시 명중
지형 회복: 병영 = 턴당 병력 10%, 마을 = 병력+책략치 회복
아이템 보정: 소지품 중 최고 1개만 적용 (중첩 없음)
```

## 8. 시나리오 챕터 (MAIN.EXE 문자열, DOS판 = 유비전 전체)

서장(황건) → 동탁타도군 일어나다 → 계교 전투 → 북해·서주 원군 → 성난 이리(여포) → 원술 토벌전 → 소패 공방전 → 관도 → 신야·박망(복재) → 와룡공명의 출려 → 조조의 남방정벌(적벽) → 형주남부 평정전 → 형주영유권 분쟁 → 익주공략전 → 한중공방전 → 촉한건국 → 형주탈환 → 중원의 사투 → 최종결전

시나리오 파일은 장 단위 5개(SNR0~4)로 분할, 스테이지 데이터는 LS11 청크(총 18개) = 우리 1부(27스테이지)가 원작 전반부+α를 더 잘게 쪼갠 구성임을 확인.

SNR*D 청크 내부: 선두에 u16 LE 오프셋 테이블(0xFFFF 종료) → 섹션별 이벤트 스크립트(바이트코드 VM). 참전 아군 목록(장수 번호 나열), 부대 배치, 증원 트리거가 스크립트 안에 인라인 — 완전 디코딩은 미착수.

**청크 = 막(幕) 구조 확정**: 일본 위키의 시나리오 구성 "서장 1막 + 1장 5막 + 2장 4막 + 3장 5막 + 종장 3막 = 18막"이 우리 청크 수(SNR0D=1, 1D=5, 2D=4, 3D=5, 4D=3)와 정확히 일치. **청크 1개 = 전투 1개가 아니라 막 1개**이며, 한 막 안에 전투 2~5개(상호배타 분기 포함: 예. 신도↔광천 택1 → 거록↔청하 택1 → 계교 합류)가 스크립트로 묶여 있다. 분기 포함 전투 번호는 51개+연전, 실제 맵은 58개(§8-2)로 대응.
- 중국 모딩 커뮤니티에 **시나리오 에디터(rekoed, 龙吟作)**가 존재 — 시나리오는 명령어(opcode) 스크립트이며 "09 = 전장(맵) 지정" 등 일부 해석됨. 작자도 전체 opcode는 미해석. 압축 해제된 시나리오 파일을 첨부 배포하므로 우리 LS11 해제 결과와 대조 가능 (§11 출처).
- 전투별 상세(적 배치 장수/병종/레벨, 증원 조건, 승리 조건, 보물, 일기토 발동 조합)는 일본 위키에 전 전투 페이지가 존재 — 예: 호뢰관 = "여포 퇴각 시 승리, 30턴 제한, 적 7부대(장료 단병 Lv6, 여포 경기병 Lv8…), 장비vs여포 일기토(인접 발동, 승리 시 Lv+1)". 필요 시 18막 × 2~5전투 전부 크롤링해 구조화 가능.

- **턴 제한**: 30턴 (원작 표준 maxTurns)

### 일기토

| 발동 조건 | 결과 | once |
|---|---|---|
| 관우가 화웅을 공격 | 관우 승리 / 화웅 퇴각 | true |

※ 원작: "관우 단기 도전" — 자동 발동, 관우 레벨 +1 보너스 (이벤트 레벨업은 현재 엔진 미구현).

### 맵 배치 근거 (재량)

맵(56×32)은 우→좌 진격 구조. x=0 방향이 성문(G) 서측, x=55 방향이 아군 진입로.
- **아군**: 동측 개방 평지 (x=50, y=14~16) — plain 확인.
- **화웅**: 성문 서측 내부 (x=2, y=14) — plain, 관문 바로 안쪽.
- **조잠**: (x=2, y=12) — plain, 화웅 북측.
- **이숙**: (x=2, y=15) — plain, 화웅 남측.
- **호진**: (x=1, y=14) — plain, 화웅 바로 서쪽.
- 전 유닛 moveCost.default < 99 검증 PASS (data.test.ts 자동 검증).

## 9. BGM 20트랙 (MUSIC.R3 인덱스 + MAIN.EXE 트랙명)

이벤트 / 프롤로그 / 마을안 / 무장사망 / 행군이동 / 전투패배 / 개인이동 / 동료가입 / 전투승리 / 에필로그 / 조조군 / 조비 / 적습 / 출진 / 성내 / 여포군 / 범용 / **결투** / 도적 (+1)

## 10. 그래픽 기술 노트

### LS11 압축 (코에이 표준, 영걸전~조조전 공용)
```
0x000  매직 "LS11" + 0 패딩 (16B)
0x010  사전 256B (빈도순 바이트 테이블)
0x110  청크 테이블: (압축크기 u32 BE, 원본크기 u32 BE, 절대오프셋 u32 BE) 반복, 압축크기 0 = 종료
       압축크기 == 원본크기 → 무압축 저장
데이터  비트스트림(MSB-first):
       code = (1이 n개 + 0) 비트열 그대로의 값 + 추가 (n+1)비트  ← 접두 가변길이 부호
       code < 0x100 → 사전[code] 1바이트 출력
       code ≥ 0x100 → 거리 = code-0x100, 길이 = 다음 code + 3 (LZ 복사)
```
구현: [tools/hero-extract/ls11_extract.py](../../tools/hero-extract/ls11_extract.py) (검증 완료 — 전 파일 해제 성공). 알고리즘 출처: [yinchuan/LSReader](https://github.com/yinchuan/LSReader) (조조전 리버싱).

### 스프라이트 픽셀 포맷
- 16색 EGA 4플레인, 1플레인 = 1bpp 행우선. 청크 내 **플레인-메이저** 배치: `[전체 스프라이트의 p0][p1][p2][p3]`
- SSCCHR2(이벤트 캐릭터) = 32×32 — **145장 추출 성공**: `tools/hero-extract/out/sprites/event_chars_v2.png`
- HEXBCHR(전투 유닛)/HEXICHR(대형)/HEXZCHR(필드)는 동일 플레인 구조이나 **타일 재조합 테이블(MAIN.EXE 내)이 별도 존재**하는 것으로 추정 — 미해독
- FACEDAT.R3 (얼굴 240장, 64×80 16색, 인덱스 = 장수 레코드 +14 필드): **비트맵이 아니라 "그리기 명령어" 시퀀스로 저장** (중국 모딩 커뮤니티 확인) — 일반 압축 해제로 안 풀린 이유. 디코딩하려면 명령어 VM 해석 필요 — 미착수
- **막힌 그래픽의 대안**: 중국 커뮤니티 범용 추출기 **RPGViewer**가 영걸전 .R3 이미지 추출(BMP/PNG)을 공식 지원 — 얼굴·유닛이 필요해지면 이걸 쓰는 게 빠름. 또는 DOSBox 스크린샷
- 팔레트: 기본 EGA 16색으로 근사 렌더 중 (원본 VGA 팔레트는 EXE 내장 추정, 미추출)

## 12. 우리 게임 매핑 노트 — packages/data/json/ 에 어떻게 쓸 것인가

### 12-1. 장수 스탯 (`generals.json` 예정)
- **원작 3능력치(통솔/무력/지력)를 시드 값으로 그대로 채용**한다. 1부 로스터 16명 전원의 원작 수치가 §3 전체 테이블에 있음.
- 우리 게임에서 무력=물리 공격, 지력=책략 공격·저항, 통솔=방어 보정(원작과 동일 역할)으로 매핑하면 공식까지 호환된다.
- 주의: 유비는 "무력 75/통솔 91" — 군주계 디자인(인덕 패시브) 시 원작처럼 통솔형으로.
- 100 능력치는 관우(통솔)·여포(무력)·제갈량(지력) 3개뿐 — **"100 = 시리즈 아이콘"** 원칙 유지 권장.

### 12-2. 전투 공식 (`combat engine`)
- §6 공식을 엔진 코어 1차 구현체로 채택 권장. 특히:
  - **보정식 4000÷(140−x)**: 능력치 인플레 없이 명장 가치를 만드는 검증된 비선형 커브. 그대로 차용.
  - **사기 = 공/방 직접 가산**: 우리 장비 "포효"(사기 저하), 일기토 사기 효과의 수치 기반이 됨.
  - 상성 75%/125%: CLAUDE.md §7 상성(기>보>궁>기)과 동일 — 계수도 채용.
- 통상 공격 명중 100% (회피 없음) — 퍼즐성(계산 가능성)의 근간. 우리도 유지 권장 (조운 "단기필마" 회피는 예외 설계로).

### 12-3. 병종 (`unit-classes.json`)
- 원작 공/방 기초치(§5)를 우리 10병종에 이식:
  - 보병(탱커) ← 장병/전차 라인 (방어 우위 120/160)
  - 기병(딜러) ← 경기병/중기병 (공격 우위, 이동 6)
  - 궁병 ← 궁병/연노병 (간접 2칸, 무반격)
  - 적병 ← 산적 라인 (산악 자유 이동 — "산악 특화" 패시브의 원형)
  - 풍수사/도사/책사 ← 주술사 책략 분담(화/수/산/교란/회복)을 3분할
- 이동력 밴드: 3(공성)/4(보병·궁병)/5~6(기병) — 우리 맵 스케일에 그대로.
- 승급: 원작 3단계 → 우리 2단계(CLAUDE.md §7 결정 사항). 기초치 점프 폭(80→120→160, 즉 +50%씩)을 2단계용으로 재배분(80→128 정도) 권장.

### 12-4. 아이템 (`items.json`)
- 무기 보정이 **+10%~+24% 곱연산**이라는 점이 핵심 (가산 아님). 최고 무기 = +24%로 캡 — 인플레 방지 설계. 우리 장비 3단계도 %캡 방식 권장.
- "소지품 중 최고 1개만 적용" 룰 — 우리는 슬롯제(무기/방어구/보조)니 슬롯당 1개로 자연 해소.
- 보물 고유효과(옥새·둔갑천서 등 분류1)는 우리 보물 도감(§10 CLAUDE.md)의 원형. 1부 20~25개 목표와 원작 63종 중 보물군 규모(약 15종)가 잘 맞음.

### 12-5. 레벨·성장
- 책략치(MP) = (레벨+10)×지력÷40 — 지력 기반 MP 성장의 검증식. 채용 권장.
- 적장 초기 레벨 커브(§3): 1장 Lv3~6 → 최종 Lv45+. 우리 레벨캡 공식(스테이지×1.5+5: 1장 Lv6~11 → 27스테이지 Lv45)과 **거의 동일한 기울기** — 원작이 우리 공식을 교차 검증해 준다.
- 경험치: 막타 중심 + 레벨차 보정 + 퇴각(격파) 보너스 — "막타 몰아주기 육성"(CLAUDE.md §10)의 원형 그대로.

### 12-6. 사운드/연출
- BGM 슬롯 설계(§9): 상황곡(승리/패배/사망/가입/적습) + 세력 테마 + 결투곡 구성을 우리 6~8트랙 + 변형 체계의 슬롯 정의로 차용.
- 영걸전(유비전 원작)에는 시스템 일기토가 없고 이벤트 한정(결투 BGM 존재) — 우리 일반 일기토(확률 발동)는 조조전 문법이므로 이 문서가 아닌 조조전 자료로 별도 보강 필요.

### 12-7. 시뮬레이션 파이프라인 입력값
- AI 자동 플레이 시뮬레이터(CLAUDE.md §11)의 **초기 파라미터 셋 = 이 문서의 §5+§6**. 원작 수치로 시작 → 우리 맵/스테이지에 맞춰 수렴 루프를 돌리는 것이 0에서 튜닝하는 것보다 빠르고 안전.
- 적 AI 행동 타입 7종(§5)·30턴 제한·반격 배율 0.5는 우리 시뮬레이터의 적 행동 모델/턴 상한 기본값으로 직행 가능. 페이싱 커브 기반 적 배치(CLAUDE.md §11-1)의 검증 대상도 원작 배치(일본 위키 전투별 페이지)와 대조하면 된다.
- 맵 칸수 기준(§8-2 통계): 페이싱 커브의 경로 거리 정규화 시 원작 평균(57×39)과 우리 목표(30×24~40×30)의 비율을 적 전력 밀도에 반영할 것.
