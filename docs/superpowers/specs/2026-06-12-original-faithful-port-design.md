- 상태: 승인됨

## 전략 결정 (사용자 확정)

**"장수 능력치·맵·병종·전투 공식을 영걸전 원작과 동일하게 먼저 만들고, 차별화는 그 위에서 한다. 아트/디자인만 다르게 간다."**

- 검증된 원작 밸런스를 통째로 이식해 0에서 튜닝하는 비용을 제거한다
- v0 엔진의 임의 추정 수치(hp/atk/def, 분산 ±10%, 상성 1.3배)는 전부 폐기
- 차별화(병종 재편 10종+2단계, 모바일 맵 축소, 병종 패시브 등 CLAUDE.md 기존 결정)는 원작 재현 검증 후 별도 단계의 백로그로 이동

## 범위

### 포함
1. **스키마 v2** — 원작 전투 모델로 데이터 모델 전면 개정
2. **변환기** `tools/import-hero/` (TS) — C:\HERO 바이너리 직독 → JSON 생성
3. **엔진 공식 교체** — applyAction 아키텍처 유지, 수치 계산만 원작 공식으로
4. **사수관 원작판 스테이지** — 56×32 원작 맵 + 위키 기반 적 배치
5. **검증** — 공식 단위 테스트(문서 검증 수치 고정) + 시뮬 베이스라인
6. **하우스키핑** — 레퍼런스 문서·추출 스크립트 커밋(코에이 에셋 제외), CLAUDE.md 갱신

### 제외 (이후)
책략 시스템(MP 소비·성공률), 경험치/레벨업/승급 실행, 사기 변동 규칙, 아이템 드랍/상점, 사수관 외 57개 맵의 스테이지화, 렌더러

## 확정 결정 4건

| 결정 | 선택 | 근거 |
|---|---|---|

## 1. 스키마 v2 (`packages/data`)

- **장수**: `{ id, name, leadership(통솔), war(무력), intelligence(지력), faceId }` — 1~100. hp/atk/def 직접 스탯 폐기
- **병종 19종** (원작 코드 0x00~0x12 순서 유지): `{ id, name, baseAtk, baseDef, move, rangeMin, rangeMax, terrainClass, promotionLine, tier }`
  - 사거리: 직접=1, 궁병·연노병=2, 발석차=3 (간접은 무반격)
  - terrainClass: `cavalry`(평지류만) / `foot`(+숲) / `all`(전 지형) 등 진입 등급
  - 승급 라인: 보병(단병→장병→전차), 궁병(궁병→연노병→발석차), 기병(경기병→중기병→친위대), 적병(산적→흉적→의적)
- **지형** (원작 테이블): `{ id, name, guard(0~0.3), moveCost(병종 카테고리별), heal? }` — 통행 불가는 코스트 99 관례 유지. 병영=병력 10%/턴, 촌락=병력+MP 회복
- **아이템 63종**: `{ id, name, category(weapon|treasure|attackItem|supplyItem|horse|book), power(b13), bonusPercent(b14) }` — 무기/병법서는 % 곱연산 보정, "소지품 중 최고 1개만 적용"
- **유닛 상태(엔진)**: `troops`(병력, 최대 보존), `morale`(사기, 기본 100), `level`, `mp`(=(레벨+10)×지력÷40), 장수 능력치 참조
- **스테이지**: 맵 파일 참조 + 배치 `{ commanderId, classId, level, troops, items[], side }` + 승리/패배 + 일기토 이벤트 (v0 구조 계승)
- **전투 설정**: `{ advantage: 0.75, disadvantage: 1.25, counterRatio: 0.5, maxTurns: 30 }` — 상성 방향: 기병>보병>궁병>기병

## 2. 변환기 (`tools/import-hero/`, TypeScript)

- `ls11.ts` — LS11 해제 (레퍼런스 §10 알고리즘, Python 구현 검증 완료분 포팅)
- `bakdata.ts` — BAKDATA.R3: 구획 C(21B×384 장수) + B(16B×63 아이템) + D(18B×384 초기 편성: 병종/레벨/병력/소지품) 파싱
- `hexzmap.ts` — HEXZMAP.R3: `[W][H][타일 W×H][속성 2bit]` → 맵 JSON. 타일 인덱스→지형 매핑 테이블은 추출 PNG(`tools/hero-extract/out/maps/`)와 대조해 수동 정의 1회, 변환기 소스에 상수로
- 출력: `packages/data/json/commanders.json`(384명), `items.json`(63종), `maps/00-sishuiguan.json` (+ 옵션으로 58개 전부)
- 실행: `pnpm import-hero --hero-dir C:\HERO`
- **코에이 바이너리·그래픽·음악은 리포에 들어가지 않는다. 수치 JSON(시스템 영역)만 커밋**

## 3. 엔진 공식 교체 (`packages/engine`)

```
공격력 = (병종공격력 + 사기 + 보정무력) × (10 + 레벨) ÷ 10 × 아이템보정
방어력 = (병종방어력 + 사기 + 보정통솔) × (10 + 레벨) ÷ 10 × 아이템보정
보정능력치 = 4000 ÷ (140 − 원능력치)      # 100→100, 90→80, 80→67, 60→50, 40→40
데미지 = max(0, 공격력 − 방어력 × 상성계수 ÷ 2) × (1 − 지형guard)
반격 데미지 = 동일 공식 × 0.5
```

- **명중 100%, 분산 없음** — v0의 varianceRatio 제거. 시드 RNG 인프라는 유지(향후 책략 성공률·크리티컬용), 통상 공격은 미사용
- **사기 = 고정 100 (이번 버전)**: 공/방 가산 필드로 구현하되 변동 규칙(피격 감소량)은 미해독. 일기토·책략의 사기 효과 도입 시 변동 로직 추가 — 이는 TBD가 아니라 명시적 단계 결정
- 이동: terrainClass 기반 진입 제한 (기병 숲 불가 등) — moveCost 99 메커니즘 재활용
- 유지: applyAction 구조, 이벤트 체계, 일기토 트리거, 퇴각 룰(병력 0=퇴각), 페이즈/승패 판정, 좌표 정렬 결정론
- 아이템 보정: 배치 데이터의 소지품에서 무기 % 최고 1개 적용 (장비 슬롯 시스템은 차후 — 원작도 소지품 자동 적용)

## 4. 사수관 스테이지 (원작판)

- 맵: 변환기 출력 56×32
- 적 배치·승리 조건·턴 제한·일기토·보물: **일본 위키(wikiwiki.jp/rimei) 사수관 전투 페이지 기반** (시나리오 바이너리 SNR*D는 미해독 — 공략 자료가 현존 최선의 원본). 조사 결과는 `docs/reference/yeonggeoljeon-data.md`에 §8-3으로 추가 기록
- 아군: 원작 1장 시점 로스터 (유비/관우/장비 등 + 초기 편성 구획 D의 레벨·병력)
- 기존 12×12 사수관(`05-sishuiguan.json`)은 원작판으로 교체 (v0 데이터는 git 이력에 남음)

## 5. 검증

- 공식 단위 테스트를 레퍼런스의 검증 수치로 고정: 보정식 5점 (100/90/80/60/40), 상성 0.75/1.25, 반격 0.5, 지형 guard, MP 공식
- 기존 엔진 테스트 36개 개정: 구조 테스트(불변성/이벤트/페이즈/일기토) 유지, 수치 기대값 교체
- 데이터 테스트: 변환기 출력 384명이 레퍼런스 발췌 표와 일치하는 스팟 체크 (관우 100/98/80, 여포 무력 100 등)
- 시뮬: maxTurns 30 (원작 표준), 그리디 정책 유지 → 사수관 베이스라인 승률 리포트 (목표치 없음 — 기록용)

## 리스크와 대응

| 리스크 | 대응 |
|---|---|
| HEXZMAP 타일 의미가 일부 미해독 (2bit 속성) | 지형 매핑은 시각 대조로 충분. 2bit 속성은 무시하고 진행, 이상 발견 시 레퍼런스 문서에 기록 |
| 위키 기반 적 배치의 정확도 | 좌표까지는 위키에 없음 — 부대 구성·승리 조건은 위키, 배치 좌표는 맵 구조에 맞춰 합리적 배치. "원작과 동일" 범위는 수치·구성까지로 정의 |
| 전투 공식의 커뮤니티 역산 오차 | 한·일 자료 일치 확인됨(레퍼런스 §11). 시뮬 결과가 상식과 크게 어긋나면 레퍼런스 재검증 |
| 사기 변동 미해독 | 고정 100으로 시작 — 공식 구조에는 들어가 있어 규칙 해명 시 데이터만 추가 |
